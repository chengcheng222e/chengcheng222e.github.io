Skills 替代 MCP？很多人都说错了

发表于 2026-03-19 分类于 AI 本文字数： 775 阅读时长 ≈ 3 分钟

最近媒体在密集传播一个观点：Skills 正在替代 MCP。

你信吗？

我第一反应是——这话听起来有道理，但哪里不对劲。

就像有人说”操作手册要替代机器”。手册写得再详细，机器不在，活儿还是干不了。

Skills 到底是什么？

说白了，Skills 是一段结构化的提示词。

它告诉模型：遇到这类任务，你应该怎么思考、按什么步骤来、注意哪些边界。比你随手写一句”帮我分析这个问题”要强得多——它更像一本操作手册，有流程、有角色、有约束。

但本质上，它还是提示词。

它的能力来源，是模型自身。

这一点很关键，也是后面所有判断的基础。

Skills 有一个天花板

Skills 越来越强，但它有一条线，过不去。

凡是需要”真实副作用”的事，Skills 永远做不到。

举四个场景：

① 实时数据
你让 Agent 查今天的股价。Skills 写得再精妙，模型没有联网能力，它只能编一个数字给你。

② 文件系统操作
读写本地文件、执行代码——这是系统级权限。提示词无法越过沙箱，这是物理边界，不是能力边界。

③ 跨系统集成
发一封邮件、在 GitHub 提一个 PR、往 Slack 发消息——这需要真实的 API 调用。”描述如何做”和”真正做到”，是两件完全不同的事。

④ 状态持久化
把结果写入数据库、保存会话状态——Skills 是无状态的提示，它不能”落地”。

一句话：Skills 能让模型更聪明地思考，但它改变不了外部世界。

MCP 补的是什么缺口

这就是 MCP 存在的理由。

MCP 是一套标准化的工具调用协议。它让模型真正连接外部世界——查数据库、调 API、读写文件、执行代码。

如果说 Skills 是”软能力”，MCP 就是”硬能力”。

两者的分工很清晰：

Skills MCP
本质 结构化提示词 工具调用协议
能力来源 模型自身 外部系统
能做什么 告诉模型怎么思考 让模型真正执行
类比 操作手册 真实的机器

所以媒体说的”替代”，其实是混淆了两个层次。

Skills 变强，是模型的”思维能力”在进化。MCP 解决的，是”真实连接”的问题。这两件事不在同一个维度上竞争。

但未来可能不是这样

现在的 Agent，大多是这样工作的：Skills 负责规划，MCP 负责执行，两者相对独立。

但有一个趋势值得关注。

当 Skills 足够强，它不只是告诉模型”怎么思考”，而是开始动态决定——什么时候调用哪个 MCP 工具、用什么参数、如何处理返回结果。

Skills 变成了 MCP 的调度层。

这时候，两者的边界会开始模糊。不是替代，而是融合。

Skills 负责”大脑”，MCP 负责”手脚”，两者深度协作，Agent 才真正完整。
